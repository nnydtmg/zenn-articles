# トーンチェック スキル

指定された記事ファイルを読み込み、筆者（nnydtmg）の確立した書き方・トーンと照合してズレを指摘する。
文体プロファイルは `/Users/horio/.claude/projects/-Users-horio-src-github-com-nnydtmg-zenn-articles/memory/writing-style.md` にも保存されている。

## 筆者の文体プロファイル

参照ベース: `articles/` 内の全記事（`aws-otel-2026.md` を除く）

### 必須要素

- **一人称**: 「私は」「私が」「私も」「私自身」を自然に使う。主語省略よりやや明示的
- **読者への語りかけ**: 「みなさん」「〜していただければ嬉しいです」「〜ご参照ください」「ぜひ」
- **謙虚さ表現**: 「〜と思います」「〜かと思います」「〜かなと思います」「〜と感じています」が文末に頻出
- **記事構造**:
  - 冒頭に `# はじめに` または `## はじめに`（なぜ書いたか、個人的背景を含む）
  - 末尾に `# 最後に` または `# まとめ`（所感や今後への期待を含む）
  - 見出しは `#` = 大セクション, `##` = 中セクション（参照ガイド的な構造より読み物的な流れ）
- **Zenn記法**: `:::message`, `:::message alert`, `:::details` を適切に使う
- **コード説明**: 実際に動かしたコードを示し、「うまくいかなかった」「手で修正した」など自分の過程も記載
- **リンク**: AWS公式ドキュメント・自分の過去記事・connpass/Speakerdeck等へのリンクを文中に自然に埋め込む
  - connpass / Speakerdeck / GitHub リポジトリのURLは、ハイパーリンクより**生URL改行貼り**スタイルを使うことが多い

### トーン特徴

- **熱量・感情**: `！` を複数使う。「最高でした」「素晴らしい」「とても良い経験」など率直な感想
- **自己開示**: ハマったこと、失敗、制約（「私もNWが本職ではないので」「かなり苦しかったです」等）を隠さず書く
- **カジュアルな合いの手**:
  - 「w」による苦笑・軽いツッコミ
  - 括弧内コメント：「（本当に助かる）」「（やってみたかった）」
  - 「。。」の多用（句点2個で「言い切れていない感」を表現）
- **励まし型の締め**: 最後の段落で読者へのエールや次のアクションを促す

### 避けるべきパターン（aws-otel-2026.md で見られるような逸脱）

- 脚注記法 `[^xxx]` —— 文中リンクに置き換える
- 記事冒頭の用語集テーブル —— 本文中で初出時に説明する
- `📌` 等の装飾絵文字をセクション見出しに使う —— Zennのcallout(`:::message`)を使う
- 参照ドキュメント的な比較表のみでの構成 —— 実際に試した経験・所感を添える
- 個人体験・経緯のない冒頭 —— 「なぜ書いたか」「どこでハマったか」が入る
- 「本記事では〜を整理します」のような報告書口調 —— 「〜してみました」「〜を試してみたいと思います」


## 実行手順

### 1. 対象ファイルの特定

ユーザーがファイルパスを指定していない場合は確認する。デフォルトは直近の会話で話題になっているファイル。

### 2. ファイルを読み込む

```
Read(file_path)
```

### 3. 各チェック項目を評価する

以下の観点ごとに「OK / 要修正 / 不明」で評価する:

| 観点 | チェック内容 |
|------|------------|
| はじめに | 個人的経緯・背景が記述されているか |
| 一人称 | 「私は/私が/私も」が自然に使われているか |
| 読者語りかけ | 「ぜひ」「〜していただければ」等があるか |
| 感情・熱量 | 感想・気づきが含まれるか（「〜と感じました」等） |
| 自己開示 | ハマりや失敗、個人的制約の言及があるか |
| カジュアル感 | 括弧内コメント・「w」・「。。」・砕けた表現があるか（過多でもNG） |
| まとめ/最後に | 所感と今後への期待・読者への言葉があるか |
| Zenn記法 | `:::message` / `:::details` が適切に使われているか |
| リンク | 公式Doc・過去記事へのリンクが文中にあるか |
| 避けるパターン | 脚注・用語集表・📌記法・報告書口調等がないか |

### 4. 結果を報告する

以下のフォーマットで出力:

```
## トーンチェック結果: <ファイル名>

### サマリ
<全体的な評価（1〜2文）>

### 要修正箇所

**[箇所1]** 行番号: XX〜YY
- 問題: <何が問題か>
- 現在: <現在のテキスト抜粋>
- 修正案: <書き直した例（筆者口調で）>

（繰り返し）

### 良い点
- <維持してほしいポイント>

### 総評
<修正の優先度と方針>
```

### 5. 修正を適用する場合

ユーザーから「修正して」と言われたら、指摘箇所を実際に Edit ツールで修正する。
修正は筆者の口調を再現するが、**内容・技術的事実は変えない**。トーンのみ調整する。
